应用简介

Regalia v1.0.4 是一款功能强大的 Android 国际象棋对弈与分析应用。本版本内置世界顶级开源棋力引擎 Stockfish 18（arm64-v8a-dotprod 原生编译），为从初学者到高级棋手提供专业级的对弈、复盘与训练体验。v1.0.4 新增 菲舍尔任意制象棋（Chess960）模式、计时赛（Time Control）、规范化 PGN 导入/导出（符合 1994 PGN 规范）、基于棋盘控制范围热力图的统计数据、Seldepth（选深）显示、视觉注解（NAG & [%csl]/[%cal]）、Web Worker 并行计算、PGN 缓存管理器、玩家重命名、中英双语 UI、认输（🏳️ Resign）等多项改进。

前台服务通知 v1.0.1 增强

当引擎进程运行时，应用会显示一个前台服务通知（Foreground Service Notification），在系统通知栏中持续展示引擎状态信息。该通知的作用是告知 Android 系统引擎进程正在执行重要计算，防止系统因内存回收或省电策略而终止引擎进程，确保对局与分析过程不被中断。

v1.0.1 简化状态：通知栏现仅显示三种引擎状态——"引擎就绪！"（就绪）、"分析中"（AI 思考 / Ponder 预判 / 复盘分析中）、"引擎错误: ..."（异常时显示截断后的错误消息）。不再显示搜索深度、节点数、速度等数值。详细的深度/节点/速度/评分仍在 APP 内顶部评估栏实时显示，通知栏保持极简风格，避免频繁更新打扰用户。
点击跳转：点击通知可跳转至 APP 主界面（PendingIntent + FLAG_ACTIVITY_NEW_TASK | SINGLE_TOP | CLEAR_TOP），方便用户从其他应用快速回到棋局。
不可手动移除：通知设置 setOngoing(true) + FLAG_NO_CLEAR，正常情况下无法被用户划掉。在 Android 14+（含小米澎湃OS 3）上系统可能允许用户手动划除，但前台服务本身仍持续运行，下次状态更新时通知会自动重新出现。
双语通知：前台服务通知的文本内容为双语显示，跟随应用当前的语言设置（保存在 SharedPreferences 中，由 JS 端 saveLangPref 同步）。中文模式下通知文本为中文，英文模式下为英文，与界面语言保持一致。
通知权限：在 Android 13（API 33）及以上版本中，应用首次启动引擎时会主动请求通知权限（POST_NOTIFICATIONS），以确保前台服务通知能够正常显示。如果用户拒绝授权，引擎仍可正常运行，但系统通知栏不会显示引擎状态信息，此时引擎进程被系统终止的风险会略有增加。

控制范围热力图

点击工具栏 🌈 按钮启用控制范围显示，棋盘格子将根据双方控制强度动态着色。热力图使用 HSL 色彩空间，通过色相（Hue）、饱和度（Saturation）和明度（Lightness）三个维度编码控制信息。

热力图着色算法

热力图颜色由三个参数决定：

  • 优势方向（adv）：我方控制数 − 对方控制数 ÷ 总控制数。adv > 0 为我方优势，adv < 0 为对方优势。决定色相（Hue）
  • 控制强度（str）：总控制数 ÷ 8（封顶 1.0）。决定饱和度（Saturation）和明度（Lightness）
  • 无人控制：双方均无控制时，使用固定色 #3a2020

实现原理：控制范围热力图

game-logic.js 中的 getCtrlMap() 遍历棋盘所有格子，对每个有棋子的格调用 attacked() 获取其攻击目标，汇总成 8×8 控制矩阵。矩阵每个元素记录 {white: [{piece, position}], black: [{piece, position}]}。

ui.js 中的 _getSqBg() 根据控制矩阵计算 HSL 颜色覆盖棋盘格背景；_updateArrows() 在 SVG 叠加层绘制控制箭头，使用持久化 SVG 元素避免重复创建 DOM。重型计算（如统计页面的全对局热力图遍历）通过 worker-pool.js 委托给 Web Worker，避免阻塞 UI。

一、开局三大黄金法则（奠定胜局基石）

  1. 中心控制：尽早控制中心格(d4,d5,e4,e5)，最大化子力效能，限制对手发展空间
  2. 快速出子：尽快出动轻子（马、象）投入战斗，避免重复走子，尽早王车易位确保王安全

二、兵型与空间管理（位置战的灵魂）

  1. 兵链完整：避免孤兵、叠兵和落后兵，维持相互保护的兵链以确保阵地稳固
  2. 空间优势：用兵链向前推进挤压对手空间，使其子力互相干涉、协调困难
  3. 通路兵：制造并推进通路兵（尤其是连接通路兵），它是中残局的终极武器

  1. 子力协调：棋子间相互支援、合力攻击同一目标，避免单子盲目深入而无后援
  2. 开放线：迅速用车（或双车重叠）占领开放线和半开放线，入侵敌方底线

English Chess Principles

I. Three Golden Opening Rules (Foundation of Victory)

  1. Center Control: Seize center squares (d4, d5, e4, e5) early to maximize piece efficiency and restrict the opponent's development
  2. Piece Development: Develop minor pieces (Knights, Bishops) quickly into battle; avoid moving the same piece twice; castle early to safeguard the King
  3. King Safety: Do not push kingside pawns (f, g, h) prematurely; castle early to tuck the King behind the pawn shield

II. Pawn Structure & Space Management (Soul of Positional Play)

  1. Pawn Structure: Avoid isolated, doubled, and backward pawns; maintain a mutually protective pawn chain to secure your position
  2. Space Advantage: Advance your pawn chain to squeeze the opponent's space, causing their pieces to interfere with each other
  3. Passed Pawns: Create and advance passed pawns (especially connected passed pawns) — they are the ultimate weapon in the middlegame and endgame

III. Piece Play & Coordination (Tactical Edge)

  1. Piece Coordination: Ensure pieces support each other and attack the same target; avoid sending a single piece deep without backup
  2. Open Files: Quickly occupy open and semi-open files with Rooks (or doubled Rooks) to invade the opponent's back rank

IV. Middlegame Strategy & Judgment (Advanced Wisdom)

  1. Flank Breakthrough: When the center is locked, launch a pawn break on the flank where your pawn chain points
  2. Exchange Principle: Simplify into a winning endgame when ahead; keep the position complex to find counterplay when behind

第十二章
菲舍尔任意制象棋（Chess960）

菲舍尔任意制象棋（Fischer Random Chess，又称 Chess960 / FR Chess）由前世界冠军鲍比·菲舍尔（Bobby Fischer）于 1996 年提出，旨在消除开局理论记忆的优势，让对局从第一步起即考验棋手的实战创造力。其核心创新是：将后排棋子的起始位置在 960 种合法排列中随机选取，但保留双方对称性、保留王车易位后的标准终点（王终至 g1/c1，车终至 f1/d1）。

960 种起始位置的命名

每种合法起始位置以 SP-ID（Starting Position Identifier） 编号 0..959 标识。SP-ID 518 即为标准国际象棋的起始位置（RNBQKBNR），用于回归测试与对照。

SP-ID白方底排排列说明
0BBQNNRKR象在同色格的最左排列之一
518RNBQKBNR标准国际象棋起始位置（回归测试基准）
959RKRNNQBB编号最大的一种排列

合法起始位置的约束

Chess960 的合法起始位置必须满足以下四条规则（共生成 960 种排列）：

  • 双象异色格：两个象必须分别位于浅色格和深色格（避免同色格双象）
  • 双车夹王：王必须位于两个车之间（不能在最左或最右）
  • 对称性：黑方底排排列与白方镜像（如白方 BBQNNRKR 对应黑方 bbqnnrkr）
  • 棋子总数：每方底排恰好 1 王、1 后、2 车、2 象、2 马

SP-ID 选择方式

新游戏对话框中开启 Chess960 后，下方显示"菲舍尔任意制象棋"专属设置区，提供以下三种方式选择 SP-ID：

  1. 🎲 随机按钮：从 0..959 中均匀随机选取一个 SP-ID（推荐，最纯粹的 Chess960 体验）
  2. 手动输入：在文本框中输入 0..959 范围内的整数
  3. 预览：底排棋子图示实时显示当前 SP-ID 的起始排列（SP-ID 518 显示 ♖ ♘ ♗ ♕ ♔ ♗ ♘ ♖）

Chess960 王车易位规则 v1.0.4 Rev34 修复

Chess960 的王车易位规则是初学者最易混淆的部分。核心原则：王车易位后王与车的终点与标准象棋完全相同——短易位（O-O）王终至 g1，车终至 f1；长易位（O-O-O）王终至 c1，车终至 d1。但起始位置因 SP-ID 不同而变化。

项目标准象棋Chess960
王起始列固定 e1（列 4）SP-ID 决定（b1-g1 任意列）
短易位车起始列固定 h1（列 7）SP-ID 决定（王右侧最近的车）
长易位车起始列固定 a1（列 0）SP-ID 决定（王左侧最近的车）
王终点g1 / c1g1 / c1（与标准象棋相同）
车终点f1 / d1f1 / d1（与标准象棋相同）

易位合法性条件

  1. 王与车均未移动过（易位权完整）
  2. 王与车之间无任何棋子（清空路径）
  3. 王当前所在格不被将军（不能在将军中易位）
  4. 王易位路径上的所有格子不被对方攻击（王不能"经过"被攻击的格子）
  5. 王易位后的目标格不被对方攻击（不能"落入"将军格）
  6. Rev34 关键修复：王路径与车路径的并集中，除王和车各自的起始位置外，每个格子必须为空——这是 Chess960 特有约束，因为王与车可能起始位置很近（如王在 b1，车在 c1），王的路径（b1→g1）会经过 d1、e1、f1 等可能被其他棋子占据的格子
Rev34 关键 bug 修复：原实现仅检查王与车之间的格子是否为空，但未检查王的目的地和车的目的地是否有其他棋子。在 Chess960 中，王和车可能起始位置很近（如王在 b1，车在 c1），王的路径（b1→g1）远远超出车的位置——经过 d1、e1、f1 等可能被其他棋子占据的格子。makeMv() 会直接覆盖这些格子，等于销毁己方棋子。修复后检查王路径和车路径的并集。

ui.js — 计时赛扣时规则（伪代码） function applyMoveTimeControl(color, moveDurationMs) { const clk = gameClocks[color]; switch (timeControlMode) { case 'untimed': // 无操作 break; case 'sudden_death': // 每步按实际用时扣时 clk.remaining -= moveDurationMs; break; case 'fischer': // 每步走完后加秒 clk.remaining -= moveDurationMs; clk.remaining += incrementMs; break; case 'bronstein': // 延迟期内不扣时，延迟期后按实际用时扣时 if (moveDurationMs <= delayMs) { // 整步在延迟期内，不扣时 } else { clk.remaining -= (moveDurationMs - delayMs); } break; case 'us_delay': // 延迟期"使用即扣"——延迟期开始即从总时长扣除， // 未用完的延迟时间不返还 const effective = Math.max(0, moveDurationMs - delayMs); clk.remaining -= effective; break; } // 超时检测 if (clk.remaining <= 0) { clk.remaining = 0; _onGameClockExpired(color); } }

时钟到期处理 v1.0.4 Rev35

Rev35 修复了时钟到期未发送 stop 指令的严重 bug。原实现 _onGameClockExpired() 仅设置 gameOver 状态，未向引擎发送 stop 指令。引擎继续基于内部 wtime 估算搜索，可能远远超过 GUI 的 0 秒标记。修复后 _onGameClockExpired() 现在调用 AndroidBridge.engineStop() 立即硬停止引擎，同时清除 isAIThinking 和 AI 安全计时器。

Rev35 关键修复：JS 调用 engineGoTimed() 时测量时钟，但 "go" 指令实际发送时（经过 stopAndWaitForBestmove + ucinewgame + readyok 等待，最长 6 秒），真实时钟已经减少。引擎基于过时（偏大）的 wtime 过度分配搜索时间。修复：engineGoTimed() 现在记录 JS 调用时间戳，在发送 "go" 前从 wtime/btime 中扣除设置开销（setup overhead），并添加安全余量（engineMoveOverhead，最小 50ms）确保引擎的 bestmove 在 GUI 时钟归零前返回。

Ponder 计时赛合规性 v1.0.4 Rev31

原实现 startPonder() 仅发送 "go ponder" 而不带时间参数——违反 UCI 规范。UCI 规范要求 go ponder 必须包含 wtime/btime/winc/binc 参数，以便引擎在收到 ponderhit 时能切换到正常计时搜索。不带时间参数会导致：(a) 引擎在 ponderhit 后无限搜索（无时间压力），或 (b) 引擎在 ponderhit 后瞬间返回 bestmove（panic 模式）——两者均为计时赛异常行为。Rev31 修复：startPonder() 新增 4 个时间参数；JS 侧传入当前 gameClocks 值。对于非计时赛，传入全 0，Java 侧检测后发送 "go ponder"（不带时间参数）。

PGN 导入

Regalia 支持通过 🗃️ 导入按钮导入 PGN（Portable Game Notation）棋谱，完整还原对局过程并在走法记录中显示变例。PGN 导入功能提供三种导入方式：

  • 粘贴 FEN / Paste FEN：从剪贴板粘贴 FEN 字符串，快速还原任意局面。适用于从其他棋类应用或网站复制的局面编码
  • 粘贴 PGN / Paste PGN：从剪贴板粘贴完整的 PGN 棋谱文本。支持标准 PGN 格式，包含头部标签（如 [FEN "..."]）、走法、注释和变例
  • 📂 选择 PGN 文件 / 📂 Select PGN File：通过 Android SAF 文件选择器从设备中选择 .pgn 文件。自动识别 FEN 和 PGN 格式

PGN 变例支持

导入的 PGN 棋谱中包含的变例（即括号内的走法，RAV — Recursive Annotation Variation）会自动在走法记录中显示为 🌿线，与引擎分析线采用相同的展示格式。变例仅显示为文本走法序列，不会触发引擎评估或在棋盘上显示，确保界面简洁清晰。

v1.0.2 修复：修复 PGN 变例"片段不完整"问题。此前若一条变例中间某个 SAN 写得不规范（缺歧义消除符、非标准后缀等），变例续着循环遇到首个无法解析的 token 时会 break，导致从该 token 起的所有剩余走法被全部丢弃——表现就是"🌿 变例只显示前半段"。v1.0.2 改为 continue（仅跳过该 token，继续解析后续走法），并对每个 token 统一应用 moveAlg 容错匹配。

PGN 解析流程

PGN 导入的解析流程经过精心设计，确保对各种 PGN 格式的兼容性：

  • 自动移除 BOM（字节顺序标记）和头部标签
  • 提取 FEN 头部（如有）以还原非标准起始局面
  • 移除注释（花括号内容）和 NAG 符号（$1, $2 等）
  • 在移除变例内容前先提取并保存变例，将其关联到对应的主走法
  • 逐着验证走法的合法性，遇到无法识别的走法时跳过而非中断
  • v1.0.1 递归变例树解析：每个 ( 开启一个新的变例帧，附加到父上下文中"最后一步走法"。嵌套 ( 附加到父变例中正确的走法（而非主线走法），符合 PGN 规范 §8.2.5
  • v1.0.1 变例首 token 匹配三重 fallback：Type A（同侧替代）→ Type A moveAlg 模糊匹配 → Type B（对手应着）→ Type B moveAlg 模糊匹配。处理非标准 SAN（缺少消歧符、+/# 后缀等）
  • v1.0.1 mainTokenIdx 索引追踪：即使主线某步走法无法解析被跳过，后续变例仍能正确关联到对应的主走法
  • v1.0.1 诊断日志：变例首 token 无法匹配时输出 console.warn，便于排查 PGN 输入问题

导入后的行为

PGN 导入完成后，应用会执行以下操作：

  • 走法记录同步更新，显示导入的所有走法及变例
  • 棋盘显示对局的最终局面
  • 自动请求引擎评估当前局面
  • 不会自动触发 AI 走棋，用户可自由浏览或开始新游戏
  • 支持复盘模式浏览导入的对局
  • v1.0.1 悔棋正常工作：导入后可逐步悔棋至白方首步并清空走法记录（修复了 v1.0.0 中 stateHistory 快照顺序错误导致的悔棋失效问题）

实现原理：容错解析

PGN 解析器（容错模式）：

  • 自动修复嵌套 {} 注释（规范禁止）
  • 未闭合注释（自动闭合）
  • 0-0 数字零 → O-O 字母 O（Chessbase 兼容）
  • 跳过无效 SAN 而非整体失败
  • 缺失 Result 默认为 *
  • 缺失 [SetUp "1"] 自动容错

v1.0.4 Rev16 修复：PGN 注释标签保留。parseStandardPGN 的头部标签剥离正则从 /\[[^\]]*\]/g 改为 /^\[[^\]]*\]/gm（行首锚定），不再误删注释内的 [%csl]/[%cal]/[%emt]/[%clk]/[%eval]。

第十五章
统计数据页面

统计数据页面（StatsActivity + stats.html）是一个全屏 WebView Activity，通过主界面走法记录卡的 📊 按钮或复盘工具栏的 📊 按钮进入。它独立于主对局 WebView，自带最小化的 JS 桥接，将繁重的 PGN 解析逻辑与主对局隔离。重型计算任务（热力图统计等）通过 worker-pool.js 委托给 Web Worker。

页面布局

统计数据页面从上到下依次为：

  1. 顶部标题栏：左侧"📊统计数据"标题，右侧四个按钮：💾HTML（导出整页为 HTML 文件）、🗃️导入（粘贴 PGN 或选择 PGN 文件）、🗂️复盘（关闭统计页并通知主界面进入复盘模式）、返回。
  2. 选择提示横幅（v1.0.3 新增）：仅当用户选中某步走法时显示。主线选中显示金色横幅（"已选中第 N 步"），变例选中显示蓝色横幅（"已选中变例第 N 步（已替换主线第 N 步）"），让用户清楚下方统计对应哪个具体局面。
  3. PGN 文本栏：完整渲染当前对局的 PGN 文本，含走法编号、走法（♔/♚ 等棋子符号样式与主棋盘一致）、注释 {}（深蓝斜体）、变例 ()（灰色）、NAG（$1 等，浅蓝色）、结果（1-0/0-1/1/2-1/2/*，金色加粗）。v1.0.3 起变例走法也可点击。
  4. 局面 FEN 栏（v1.0.3 新增）：位于 PGN 文本栏下方。等宽字体显示当前选中走法刚刚走出后的局面的 FEN（后台通过 stateToFEN() 自动生成）。"复制"按钮位于右下角（独立一行，flexbox 右对齐），避免与 FEN 文本重叠。无选中走法时显示初始局面 FEN + 提示文字。
  5. 对局元数据：表格显示 [Event]、[Site]、[Date]、[Round]、[White]、[Black]、[Result]、[TimeControl] 头标签内容。
  6. 局面统计：总步数、最终子力、子力差。
  7. 走法质量：白方/黑方准确率（Chess.com 风格 CPL 公式）、平均 CPL、6 级分类计数（妙手/最佳/好棋/不准/错误/大错）。仅在有评估数据时显示。
  8. 评估趋势图：SVG 折线图，横轴为步数，纵轴为评估值（白方视角，单位百兵）。
  9. 子力平衡图：SVG 折线图，横轴为步数，纵轴为白方-黑方子力差。
  10. 棋子活动度：吃子数、将军数、王车易位数、升变数；按棋子类型（兵/马/象/车/后/王）分别统计白黑双方的移动次数表格。
  11. 阶段分析：基于棋盘上非兵非王棋子数量判定阶段（开局 >8、中局 4-8、残局 <4），统计各阶段的步数（ply）。
  12. 关键转折：评估变化最大的 10 步（含步号、评估变化值、分类标签）。
  13. 时间管理（v1.0.3 升级）：详见 15.3。
  14. 开局分析：前 8 步走法序列、白黑双方轻子（马/象）出子数。
  15. 兵型：叠兵/孤兵/通路兵计数（白黑分别统计）。
  16. 中心控制：白黑双方走法落在中心格（d4/e4/d5/e5）的次数。
  17. 王的安全：白黑双方将军次数、王移动次数。
  18. 局面复杂度：评估波动率（标准差）、领先转换次数、白方/黑方优势步数。
  19. 热力图控制统计（v1.0.4 新增）：8×8 平均控制热力图，每格显示 (白方平均 / 黑方平均) 攻击子数，背景色用 HSL 表示控制净值。
  20. 视觉注解（v1.0.4 新增）：NAG 分布、各色箭头/格子计数。

时间管理分析（v1.0.3 升级）

v1.0.3 起时间管理模块从 v1.0.2 的"仅显示 [%clk] 计数"升级为完整的时间分析。新增 extractMoveTimes() 函数自动识别 PGN 注释 {} 内的用时数据，支持以下格式（按识别优先级）：

  1. v1.0.3 直述格式：{3.2s +0.5 略优}（用时 + 空格 + 评估）。这是 v1.0.3 起 APP 自身导出 PGN 时使用的格式。extractMoveTimes() 取注释内第一个 数字s 模式作为该步用时。无需任何头部。
  2. 标准 PGN 时钟格式（需 [TimeControl] 头）：[%clk HH:MM:SS] 表示走完该步后该棋手剩余时钟时间。APP 通过 [TimeControl "BASE+INC"] 头推算每步实际用时。支持 "7200+60"（基础+增量）、"1800"（仅基础，无增量）、"40/7200"（40 步内 7200 秒，按无增量处理）三种 TimeControl 写法。无 [TimeControl] 头时，[%clk] 仅记录时钟剩余值，不推算用时。
  3. 早期破折号格式：{3.2s——+0.5 略优}（用时 + 破折号 + 评估）。向后兼容——早期 v1.0.3 测试版导出的 PGN 仍能正确解析。

  • 平均每步 / 白方平均 / 黑方平均：每步平均用时
  • 最长用时步：含步号（如"5."或"5..."）与用时
  • 最短用时步：含步号与用时

点击变例走法查看变例统计（v1.0.3 新增）

v1.0.2 仅主线走法可点击选中查看逐局面统计。v1.0.3 起 () 内的变例走法也可点击。点击变例走法时，统计页按以下规则构造合成走法序列：

  1. 取主线走法 [0..mainIdxBefore-1]，其中 mainIdxBefore 为 () 左侧相邻主线走法的索引（即被变例首步替换的那步走法）。
  2. 追加变例走法 [0..clickedIdx]（从变例首步到被点击的那一步）。变例走法通过 applySANMove() 重新解析（支持王车易位、升变、吃子、消歧义等）。
  3. 用合成序列重新计算全部统计模块。

例如 PGN 1. e4 e5 2. Nf3 (2. Bc4 Nf6 3. d3) Nc6 中，用户点击变例内的 3. d3：

  • mainIdxBefore = 3（即主线第 4 步 Nf3，索引 3）
  • 合成序列 = 主线前 3 步 [e4, e5, Nf3] + 变例前 3 步 [Bc4, Nf6, d3] = 6 步
  • 统计模块显示 3.d3 走完后的局面统计

局面 FEN 栏（v1.0.3 新增）

局面 FEN 栏位于 PGN 文本栏下方，显示当前选中走法刚刚走出后的局面的 FEN 字符串。FEN 由后台 stateToFEN() 函数自动从选中走法的 postState 生成，包含 6 个字段：

  1. 棋盘摆放（rank 8 → rank 1，/ 分隔，空格用数字表示）
  2. 轮走方（w 或 b）
  3. 王车易位权（KQkq 或 -，Chess960 使用 Shredder-FEN 格式如 HAah）
  4. 吃过路兵目标格（如 e3 或 -）

Ponder 预判实现机制

Ponder（预判）功能基于 UCI（Universal Chess Interface）协议的 go ponder 命令实现。本章节详细描述完整数据流、四层防护机制与计时赛合规性。

完整数据流

  1. 引擎返回最佳走法：引擎通过 stdout 输出 bestmove e2e4 ponder e7e5，其中 e2e4 是推荐走法，e7e5 是引擎预测对手的应手。
  2. Java 层解析：StockfishNative.processEngineLine() 使用正则 ^bestmove\s+(\S+)(?:\s+ponder\s+(\S+))? 解析出 ponder 走法，存入 _lastPonderMove 字段。若无 ponder 字段则置 null。
  3. JS 层获取 ponder 数据：onBestMove() 和 onHintMove() 均通过 AndroidBridge.getLastPonderMove() 获取当前 ponder 走法。getLastPonderMove() 为一次性读取（读取后自动清空），防止旧数据泄漏到后续搜索。获取到的 UCI 走法暂存于 _pendingPonderMoveUCI，等待首个 onPonderProgress() 回调时再转换为 SAN 格式并赋值 _ponderMoveSAN。
  4. 启动 ponder 搜索：AI 走棋后，onBestMove() 将 ponder 走法应用到当前局面（通过 makeMv() 模拟对手走棋），生成新 FEN，调用 AndroidBridge.startPonder(fen)。Java 层向引擎发送 position fen <FEN> 和 go ponder 命令，引擎进入无限搜索模式。
  5. Ponder 进度回调与延迟转换：引擎搜索时通过 info 行实时回调 onPonderProgress()，更新 _ponderBarInfo（搜索深度、节点数、评估值）。首次回调时，若 _pendingPonderMoveUCI 存在，则将其转换为 SAN 格式赋值 _ponderMoveSAN（一次性，转换后清空）。此延迟机制确保预判走法和搜索进度在同一时刻出现，避免 UI 显示不完整。
  6. Ponder Hit / Stop：玩家走棋时，executeMove() 检查是否匹配 ponder 走法。匹配则调用 AndroidBridge.ponderHit()（引擎利用已有搜索结果继续分析）；不匹配则调用 AndroidBridge.stopPonder()（引擎停止搜索）。
  7. 显示条件：AI 对手栏第三行仅当同时存在 _ponderMoveSAN（SAN 格式的预判走法）和 _ponderBarInfo（搜索进度）时才显示 🔮 预判信息。当 AI 正在思考、提示加载中、复盘模式或游戏结束时，自动隐藏。

四层防护机制

Regalia 采用四层防护确保 Ponder 数据不会残留过期信息：

  1. 世代计数器 / Generation Counter：JS 层核心防线，所有 onPonderProgress() 回调必须通过世代校验才能更新显示数据。每次启动新的 ponder 会话时 _ponderGen++，回调时若 _ponderGen !== _ponderStartGen 则丢弃。
  2. 一次性读取 / One-time read：getLastPonderMove() 读取后自动清空，防止旧 ponder 走法泄漏到后续搜索。
  3. 即时 DOM 更新 / Immediate DOM update：清除 ponder 变量时立即调用 _updateAIThinkDisplay()，避免变量已清除但 DOM 仍显示旧值。
  4. Java 层标志位 / Java-layer flag：_discardingPonderBestmove 标志防止 ponder 停止后的 bestmove 被错误路由到正常搜索处理流程。

提示场景中，onHintMove() 将 ponder 走法直接转换为 SAN 并嵌入 hintText，与推荐走法一同显示（如「🔦 推荐: Nf3 🔮 预判: e5」），无需等待 Ponder 搜索进度。条件显示：

  • AI 对手栏第三行：在引擎后台 Ponder 搜索活跃时显示预判信息
  • 无预判信息时：不会显示任何 Ponder 相关内容，保持界面简洁

Ponder 数据时效性

Ponder 预判信息始终来自最近一次引擎反馈的 bestmove 响应中的 ponder 字段。每次引擎返回新的 bestmove 时，旧的预判数据会立即清除，仅当新 bestmove 包含 ponder 信息时才会更新显示，确保预判数据不会残留过期信息。在提示场景中，ponder 走法在引擎返回 bestmove 时即被转换为 SAN 格式（基于推荐走法执行后的局面进行转换，确保着法正确性），直接嵌入提示文本，无需等待 Ponder 搜索进度回调。

防陈旧数据机制（多层防护）

  1. 每次清除 ponder 变量时立即调用 _updateAIThinkDisplay()更新 DOM，避免变量已清除但 DOM 仍显示旧值
  2. onPonderProgress() 增加代际守卫（_ponderGen / _ponderStartGen），若当前回调的代际与启动时不一致，说明 ponder 已被其他流程清除，回调应丢弃
  3. onEngineProgress() 在 AI 思考或提示加载期间主动清除残留 ponder 数据
  4. Java 层增加 _discardingPonderBestmove 标志，防止 ponder 停止后的 bestmove 被错误路由到正常搜索处理流程
  5. 提示区域现在要求 _ponderMoveSAN 和 _ponderBarInfo 同时存在才显示 🔮 预判，避免新 ponder 会话启动后但首帧进度尚未到达时显示旧 ponder 数据
