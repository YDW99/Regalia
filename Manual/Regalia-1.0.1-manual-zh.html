应用简介

Regaliav1.0.1 是一款功能强大的 Android 国际象棋对弈与分析应用。本版本内置世界顶级开源棋力引擎 Stockfish 18（arm64-v8a-dotprod 原生编译），为从初学者到高级棋手提供专业级的对弈、复盘与训练体验。

18 Stockfish 引擎版本
6+SL AI 难度等级
8 MultiPV 线数

核心特性

  • 内置 Stockfish 18 引擎，arm64-v8a-dotprod 原生编译，自动利用 NEON 加速指令集
  • 6 级预设 AI 难度 + SL档（Skill Level 自定义）+ ⚙️手动设置，适合各水平棋手
  • AI 思考进度条实时显示搜索深度、节点数、速度与评估分数
  • Ponder（预判）模式，AI 在对手回合继续思考，充电时推荐开启以加速后续走法
  • MultiPV 多线分析（1-8 线），同时展示多条引擎推荐变化
  • 变例系统：走法记录与复盘中展示引擎预测续着和备选变化（💬变例 开关控制）
  • 完整复盘模式：逐步导航、评估趋势图、走法分类与关键着标注、全部分析
  • ECO 开局库，含搜索、开局推荐与家族筛选
  • Syzygy 在线残局库查询（7子及以下残局必胜/和棋/必败判定）
  • WDL 胜和负概率显示（默认开启），直观了解局面优劣
  • 评估分数含 SF18 引擎标识徽章、搜索深度 D 标识、节点与速度信息
  • 格子控制范围热力图（🌈/🌗 切换），含箭头指示与格子控制信息面板
  • AI 提示多线展示，显示所有推荐走法及评分
  • 棋理提示面板，实时提供十大棋理原则提醒
  • 引擎配置导出/导入（key=value 文本格式），方便多设备同步
  • FEN 复制/导入，支持任意局面快速还原
  • 摆棋模式，自由编辑棋盘局面，含撤销/恢复/重置/清空
  • 兼容 Android 5.0 至 Android 15，适配 Xiaomi HyperOS 3

实现原理：引擎初始化流程

StockfishNative.java 在 onCreate() 中将 assets 中的引擎二进制文件复制到应用私有目录（context.getFilesDir()），通过 JNI 调用 nativeChmod() 赋予执行权限。WebView 加载 chess.html 后，JS 层调用 AndroidBridge.startEngine() 启动 UCI 进程，引擎通过 stdout 返回 uciok 后标记 _engineReady=true。

前台服务通知 v1.0.1 增强

当引擎进程运行时，应用会显示一个前台服务通知（Foreground Service Notification），在系统通知栏中持续展示引擎状态信息。该通知的作用是告知 Android 系统引擎进程正在执行重要计算，防止系统因内存回收或省电策略而终止引擎进程，确保对局与分析过程不被中断。

v1.0.1 简化状态：通知栏现仅显示三种引擎状态——"引擎就绪！"（就绪）、"分析中"（AI 思考 / Ponder 预判 / 复盘分析中）、"引擎错误: ..."（异常时显示截断后的错误消息）。不再显示搜索深度、节点数、速度等数值。详细的深度/节点/速度/评分仍在 APP 内顶部评估栏实时显示，通知栏保持极简风格，避免频繁更新打扰用户。
点击跳转：点击通知可跳转至 APP 主界面（PendingIntent + FLAG_ACTIVITY_NEW_TASK | SINGLE_TOP | CLEAR_TOP），方便用户从其他应用快速回到棋局。
不可手动移除：通知设置 setOngoing(true) + FLAG_NO_CLEAR，正常情况下无法被用户划掉。在 Android 14+（含小米澎湃OS 3）上系统可能允许用户手动划除，但前台服务本身仍持续运行，下次状态更新时通知会自动重新出现。
双语通知：前台服务通知的文本内容为双语显示，跟随应用当前的语言设置（保存在 SharedPreferences 中，由 JS 端 saveLangPref 同步）。中文模式下通知文本为中文，英文模式下为英文，与界面语言保持一致。
通知权限：在 Android 13（API 33）及以上版本中，应用首次启动引擎时会主动请求通知权限（POST_NOTIFICATIONS），以确保前台服务通知能够正常显示。如果用户拒绝授权，引擎仍可正常运行，但系统通知栏不会显示引擎状态信息，此时引擎进程被系统终止的风险会略有增加。

控制范围热力图

点击工具栏 🌈 按钮启用控制范围显示，棋盘格子将根据双方控制强度动态着色。热力图使用 HSL 色彩空间，通过色相（Hue）、饱和度（Saturation）和明度（Lightness）三个维度编码控制信息。

热力图着色算法

热力图颜色由三个参数决定：

  • 优势方向（adv）：我方控制数 − 对方控制数 ÷ 总控制数。adv > 0 为我方优势，adv < 0 为对方优势。决定色相（Hue）
  • 控制强度（str）：总控制数 ÷ 8（封顶 1.0）。决定饱和度（Saturation）和明度（Lightness）
  • 无人控制：双方均无控制时，使用固定色 #3a2020

实现原理：控制范围热力图

game-logic.js 中的 getCtrlMap() 遍历棋盘所有格子，对每个有棋子的格调用 attacked() 获取其攻击目标，汇总成 8×8 控制矩阵。矩阵每个元素记录 {white: [{piece, position}], black: [{piece, position}]}。

ui.js 中的 _getSqBg() 根据控制矩阵计算 HSL 颜色覆盖棋盘格背景；_updateArrows() 在 SVG 叠加层绘制控制箭头，使用持久化 SVG 元素避免重复创建 DOM。

实现原理：Ponder 预判机制

Ponder（预判）功能基于 UCI（Universal Chess Interface）协议的 go ponder 命令实现。完整数据流如下：

  1. 引擎返回最佳走法：引擎通过 stdout 输出 bestmove e2e4 ponder e7e5，其中 e2e4 是推荐走法，e7e5 是引擎预测对手的应手。
  2. Java 层解析：StockfishNative.processEngineLine() 使用正则 ^bestmove\s+(\S+)(?:\s+ponder\s+(\S+))? 解析出 ponder 走法，存入 _lastPonderMove 字段。若无 ponder 字段则置 null。
  3. JS 层获取 ponder 数据：onBestMove() 和 onHintMove() 均通过 AndroidBridge.getLastPonderMove() 获取当前 ponder 走法。getLastPonderMove() 为一次性读取（读取后自动清空），防止旧数据泄漏到后续搜索。获取到的 UCI 走法暂存于 _pendingPonderMoveUCI，等待首个 onPonderProgress() 回调时再转换为 SAN 格式并赋值 _ponderMoveSAN，确保 _ponderMoveSAN 和 _ponderBarInfo 始终同步更新，避免仅有走法而无搜索进度的部分显示。
  4. 启动 ponder 搜索：AI 走棋后，onBestMove() 将 ponder 走法应用到当前局面（通过 makeMv() 模拟对手走棋），生成新 FEN，调用 AndroidBridge.startPonder(fen)。Java 层向引擎发送 position fen <FEN> 和 go ponder 命令，引擎进入无限搜索模式。
  5. Ponder 进度回调与延迟转换：引擎搜索时通过 info 行实时回调 onPonderProgress()，更新 _ponderBarInfo（搜索深度、节点数、评估值）。首次回调时，若 _pendingPonderMoveUCI 存在，则将其转换为 SAN 格式赋值 _ponderMoveSAN（一次性，转换后清空）。此延迟机制确保预判走法和搜索进度在同一时刻出现，避免 UI 显示不完整。
  6. Ponder Hit / Stop：玩家走棋时，executeMove() 检查是否匹配 ponder 走法。匹配则调用 AndroidBridge.ponderHit()（引擎利用已有搜索结果继续分析）；不匹配则调用 AndroidBridge.stopPonder()（引擎停止搜索）。
  7. 显示条件：AI 对手栏第二行仅当同时存在 _ponderMoveSAN（SAN 格式的预判走法）和 _ponderBarInfo（搜索进度）时才显示 🔮 预判信息。当 AI 正在思考、提示加载中、复盘模式或游戏结束时，自动隐藏。提示场景中，onHintMove() 将 ponder 走法直接转换为 SAN 并嵌入 hintText，与推荐走法一同显示（如「🔦 推荐: Nf3 🔮 预判: e5」），无需等待 Ponder 搜索进度。

Ponder 预判实现机制

什么是 Ponder？/ What is Ponder?

Ponder（预判/后台思考）是 UCI（Universal Chess Interface）协议中定义的一项功能：当引擎完成自己的走法后，可以继续在后台预测对手的应手，并提前对该预测局面进行搜索分析。如果对手实际走出的棋与引擎预测一致（Ponder Hit），引擎可立即利用已分析的结果，几乎零延迟地返回下一步走法；若不一致（Ponder Miss），引擎停止后台搜索并重新分析新局面。

Ponder (background thinking) is a feature defined in the UCI (Universal Chess Interface) protocol: after the engine completes its move, it can continue in the background to predict the opponent's reply and pre-analyze that predicted position. If the opponent's actual move matches the engine's prediction (Ponder Hit), the engine can immediately use the already-analyzed result and return the next move with near-zero latency; if not (Ponder Miss), the engine stops the background search and re-analyzes the new position.

Regalia 中的 Ponder 实现 / Ponder Implementation in Regalia

Regalia 的 Ponder 实现遵循 UCI 协议标准，完整数据流如下：

Regalia's Ponder implementation follows the UCI protocol standard. The complete data flow is as follows:

  1. 引擎输出最佳走法 / Engine outputs best move：AI 思考完成后，引擎通过 stdout 输出 bestmove X ponder Y，其中 X 为推荐走法，Y 为引擎预测对手的应手（ponder move）。若引擎未提供 ponder 字段，则 Y 为空。
  2. 解析与存储 / Parse and store：Java 层 StockfishNative.processEngineLine() 使用正则表达式从 bestmove 行中提取 ponder 走法，存入 _lastPonderMove 字段。若不存在则置 null。
  3. 启动后台 Ponder 搜索 / Start background ponder search：AI 走棋后，onBestMove() 将 ponder 走法应用到当前局面（模拟对手走出该步），生成新 FEN，调用 AndroidBridge.startPonder(fen)。Java 层向引擎发送 position fen <FEN> 和 go ponder 命令，引擎进入无限搜索模式。

  5. Ponder Hit / Stop：玩家走棋时，检查是否匹配 ponder 走法。匹配则发送 ponderhit；不匹配则发送 stop。

世代计数器（Generation Counter）防陈旧机制

Generation Counter Anti-Stale Mechanism

为防止陈旧的 Ponder 数据被错误显示，Regalia 引入了世代计数器（Generation Counter）机制。每次启动新的 Ponder 搜索时，计数器递增；所有 Ponder 回调携带当前世代标识，仅当回调的世代与当前活跃世代匹配时，数据才会被接受并显示。

To prevent stale Ponder data from being incorrectly displayed, Regalia introduces a Generation Counter mechanism. Each time a new Ponder search is started, the counter increments; all Ponder callbacks carry the current generation identifier, and data is only accepted and displayed when the callback's generation matches the currently active generation.

  • 世代递增 / Generation increment：每次 startPonder() 调用时，_ponderGeneration 递增，新的 onPonderProgress() 回调必须携带相同世代才有效
  • 陈旧回调丢弃 / Stale callback discard：若 onPonderProgress() 回调的世代不匹配当前活跃世代，该回调被立即丢弃，不会更新 _ponderBarInfo 或 _ponderMoveSAN
  • 停止时递增 / Increment on stop：调用 stopPonder() 或 ponderHit() 时同样递增世代计数器，确保停止后的任何延迟回调被自动忽略
  • 与一次性读取协同 / Synergy with one-time read：世代计数器与 getLastPonderMove() 的一次性读取机制协同工作，形成双层防护：Java 层防止旧 ponder 走法泄漏，JS 层防止旧搜索进度残留
