App Introduction

Regalia v1.0.6 is a powerful Android chess playing and analysis app. This version is built around the world-class open-source engine Stockfish 18 (arm64-v8a-dotprod native build) and provides a professional-grade playing, review, and training experience for everyone from beginners to advanced players. v1.0.4 introduces Fischer Random Chess (Chess960) mode, Time Control, standardized PGN import/export (compliant with the 1994 PGN specification), heatmap-control-based statistics, Seldepth (selective depth) display, visual annotations (NAG & [%csl]/[%cal]), Web Worker parallel computation, a PGN cache manager, player renaming, bilingual Chinese/English UI, resign (🏳️ Resign), and many other improvements. v1.0.5 adds sensor-fusion board anti-shake (OIS principle), high aspect-ratio screen adaptation, notch/cutout/R-corner adaptation, phase analysis precision (opening/middlegame/endgame multi-criteria detection), review arrow shrink optimization, and more.

Data flow between modules follows strict unidirectional dependency: ui.js calls pure functions from game-logic.js to obtain game state, communicates with the engine via ai-bridge.js, and ai-bridge.js interacts with the Java-layer Stockfish process through AndroidBridge. First Launch

On first launch, the app displays a loading screen: ♔ King icon ( #E8E8F0) + app name + progress bar ( #d4a017→#ffd700 gradient). Implementation Details: Engine Initialization Flow

StockfishNative.java copies the engine binary from assets to the app's private directory (context.getFilesDir()) in onCreate(), granting execute permission via a JNI call to nativeChmod(). After WebView loads chess.html, the JS layer calls AndroidBridge.startEngine() to start the UCI process; once the engine returns uciok via stdout, _engineReady=true is set.

Foreground Service Notification v1.0.1 Enhanced

While the engine process is running, the app displays a Foreground Service Notification that continuously shows engine status information in the system notification bar. Its purpose is to inform the Android system that the engine process is performing important computation, preventing the system from terminating the engine process due to memory reclamation or power-saving policies, and ensuring that play and analysis are not interrupted.

v1.0.1 Simplified state: The notification now shows only three engine states — "Engine Ready!" (ready), "Analyzing" (AI thinking / Ponder / review analysis in progress), and "Engine error: ..." (truncated error message on failure). Search depth, node count, and speed are no longer displayed. Detailed depth/nodes/speed/score data is still shown in real time in the in-app top eval bar; the notification is intentionally minimal to avoid frequent-update distractions.
Click-to-open: Tapping the notification opens the app's main screen (PendingIntent + FLAG_ACTIVITY_NEW_TASK | SINGLE_TOP | CLEAR_TOP), allowing the user to quickly return to the game from any other app.
Non-removable: The notification sets setOngoing(true) + FLAG_NO_CLEAR; under normal conditions it cannot be swiped away by the user. On Android 14+ (including Xiaomi HyperOS 3), the system may allow manual dismissal, but the foreground service itself keeps running, and the notification auto-restores on the next status update.
Bilingual notification: The text content of the Foreground Service Notification is bilingual, following the app's current language setting (stored in SharedPreferences, synced from JS via saveLangPref). In Chinese mode the notification text is Chinese; in English mode it is English, consistent with the interface language.
Notification permission: On Android 13 (API 33) and above, the app proactively requests Notification Permission (POST_NOTIFICATIONS) when the engine is first launched, to ensure the Foreground Service Notification can be displayed normally. If the user denies permission, the engine still runs normally, but the system notification bar will not display engine status information, slightly increasing the risk of the engine process being terminated by the system.

Control Heatmap

Tap the 🌗 toolbar button to enable control-range display (the 🌗 button becomes 🌈 when enabled; tap again to disable) — board squares are dynamically colored by the control strength of each side. The heatmap uses the HSL color space, encoding control information via three dimensions: hue, saturation, and lightness.

Heatmap Coloring Algorithm

The heatmap color is determined by three parameters:

  • Advantage direction (adv): (own control count − opponent control count) ÷ total control count. adv > 0 = own advantage; adv < 0 = opponent advantage. Determines hue.
  • Control strength (str): total control count ÷ 8 (capped at 1.0). Determines saturation and lightness.
  • No control: when neither side has any control, the fixed color #3a2020 is used.

Implementation Details: Control Heatmap

getCtrlMap() in game-logic.js walks every square on the board; for each occupied square it calls attacked() to get the piece's attack targets, aggregating them into an 8×8 control matrix. Each matrix cell records {white: [{piece, position}], black: [{piece, position}]}.

_getSqBg() in ui.js computes the HSL color from the control matrix and overrides the board-square background; _updateArrows() draws control arrows on the SVG overlay using persistent SVG elements to avoid recreating DOM. Heavy compute (such as full-game heatmap traversal on the statistics page) is delegated to a Web Worker via worker-pool.js to avoid blocking the UI.

game-logic.js — getCtrlMap() control-range calculation function getCtrlMap(b){ const cm=[]; for(let r=0;r<8;r++){ cm[r]=[]; for(let c=0;c<8;c++) cm[r][c]={white:[],black:[]}; } for(let r=0;r<8;r++) for(let c=0;c<8;c++){ const p=b[r][c]; if(p){ const atks=attacked(b,{row:r,col:c}); for(const a of atks){ cm[a.row][a.col][p.color].push( {piece:p,position:{row:r,col:c}}); } } } return cm; }
ui.js — _getSqBg() heatmap coloring algorithm function _getSqBg(rr,cc,cm,isL,lastMove){ let bg=isL?SQ_LIGHT:SQ_DARK; let ctrlBg=null; if(cm){ const e=cm[rr][cc]; if(e){ const wc=e.white.length, bc=e.black.length; const myC=playerColor==='white'?wc:bc; const opC=playerColor==='white'?bc:wc; const net=myC-opC; const total=myC+opC; const adv=total>0?net/total:0; const str=Math.min(1,total/8); let hue; if(myC===0&&opC===0){ ctrlBg='#3a2020'; // no control }else{ if(adv>=0) hue=280-adv*60; else hue=280-adv*80; if(hue>=360) hue-=360; const sat=0.50+str*0.40; // 50%~90% const lit=0.48-str*0.12; // 36%~48% ctrlBg=`hsl(${Math.round(hue)},${Math.round(sat*100)}%,${Math.round(lit*100)}%)`; } } } if(ctrlBg) bg=ctrlBg; return bg; }

Making Moves

The app uses tap-to-move: tap the piece you want to move (the piece is highlighted and legal moves are shown), then tap the target square to complete the move. When a pawn reaches the back rank, a promotion dialog appears (Queen → Rook → Bishop → Knight). In Chess960 mode, castling follows the official Chess960 rules: King ends on g1/c1, Rook ends on f1/d1, but starting positions are randomized.

The top 10 chess principles are organized under four strategic dimensions and switch automatically with the interface language: Chinese mode shows Chinese principles, English mode shows English principles.

English Chess Principles

I. Three Golden Opening Rules (Foundation of Victory)

  1. Center Control: Seize center squares (d4, d5, e4, e5) early to maximize piece efficiency and restrict the opponent's development
  2. Piece Development: Develop minor pieces (Knights, Bishops) quickly into battle; avoid moving the same piece twice; castle early to safeguard the King
  3. King Safety: Do not push kingside pawns (f, g, h) prematurely; castle early to tuck the King behind the pawn shield

II. Pawn Structure & Space Management (Soul of Positional Play)

  1. Pawn Structure: Avoid isolated, doubled, and backward pawns; maintain a mutually protective pawn chain to secure your position
  2. Space Advantage: Advance your pawn chain to squeeze the opponent's space, causing their pieces to interfere with each other
  3. Passed Pawns: Create and advance passed pawns (especially connected passed pawns) — they are the ultimate weapon in the middlegame and endgame

III. Piece Play & Coordination (Tactical Edge)

  1. Piece Coordination: Ensure pieces support each other and attack the same target; avoid sending a single piece deep without backup
  2. Open Files: Quickly occupy open and semi-open files with Rooks (or doubled Rooks) to invade the opponent's back rank

IV. Middlegame Strategy & Judgment (Advanced Wisdom)

  1. Flank Breakthrough: When the center is locked, launch a pawn break on the flank where your pawn chain points
  2. Exchange Principle: Simplify into a winning endgame when ahead; keep the position complex to find counterplay when behind

Fischer Random Chess (Chess960)

Fischer Random Chess (also called Chess960 / FR Chess) was proposed by former world champion Bobby Fischer in 1996, with the goal of eliminating the advantage of opening-theory memorization and forcing players to demonstrate practical creativity from move one. The core innovation: the back-rank pieces' starting position is randomly chosen from 960 legal arrangements, while preserving symmetry between the two sides and preserving the standard final squares of castling (King ends on g1/c1, Rook ends on f1/d1).

Naming of the 960 Start Positions

Each legal start position is identified by an SP-ID (Starting Position Identifier) numbered 0..959. SP-ID 518 is the standard chess start position (RNBQKBNR), used for regression testing and comparison.

SP-IDWhite Back-Rank ArrangementNotes
0BBQNNRKROne of the leftmost arrangements with bishops on opposite colors
518RNBQKBNRStandard chess start position (regression baseline)
959RKRNNQBBThe highest-numbered arrangement

Constraints on Legal Start Positions

A legal Chess960 start position must satisfy the following four rules (yielding 960 arrangements):

  • Bishops on opposite colors: the two bishops must be on a light and a dark square (avoiding same-color bishops)
  • King between rooks: the King must be between the two rooks (not at the leftmost or rightmost)
  • Symmetry: Black's back-rank arrangement mirrors White's (e.g. White BBQNNRKR ↔ Black bbqnnrkr)
  • Piece totals: each side has exactly 1 King, 1 Queen, 2 Rooks, 2 Bishops, 2 Knights on the back rank

SP-ID Selection Methods

When Chess960 is enabled in the New Game dialog, the "Fischer Random Chess" settings area appears below, offering three ways to choose an SP-ID:

  1. 🎲 Random button: uniformly random SP-ID from 0..959 (recommended, the purest Chess960 experience)
  2. Manual input: enter an integer 0..959 in the text box
  3. Preview: the back-rank piece diagram shows the starting arrangement for the current SP-ID in real time (SP-ID 518 shows ♖ ♘ ♗ ♕ ♔ ♗ ♘ ♖)

Chess960 Castling Rules v1.0.4 Rev34 Fix

Chess960 castling is the part most confusing for beginners. The core principle: after castling, the King and Rook final squares are exactly the same as in standard chess — kingside (O-O) King ends on g1, Rook ends on f1; queenside (O-O-O) King ends on c1, Rook ends on d1. But the starting positions vary by SP-ID.

ItemStandard ChessChess960
King start columnFixed e1 (column 4)SP-ID-decided (any column b1-g1)
Kingside Rook start columnFixed h1 (column 7)SP-ID-decided (nearest rook to the King's right)
Queenside Rook start columnFixed a1 (column 0)SP-ID-decided (nearest rook to the King's left)
King end squareg1 / c1g1 / c1 (same as standard)
Rook end squaref1 / d1f1 / d1 (same as standard)

Castling Legality Conditions

  1. Neither King nor Rook has moved (castling right intact)
  2. No pieces between King and Rook (clear path)
  3. The King's current square is not in check (cannot castle out of check)
  4. All squares on the King's castling path are not attacked by the opponent (King cannot "pass through" attacked squares)
  5. The King's destination square is not attacked by the opponent (cannot "land in" a checked square)
  6. Rev34 critical fix: in the union of the King path and the Rook path, every square must be empty except for the King's and Rook's respective starting squares — this is a Chess960-specific constraint because the King and Rook may start close together (e.g. King on b1, Rook on c1), and the King's path (b1→g1) passes through d1, e1, f1 etc. which may be occupied by other pieces
Rev34 critical bug fix: The original implementation only checked whether squares between King and Rook were empty, but did not check whether the King's destination and the Rook's destination had other pieces. In Chess960 the King and Rook may start close (e.g. King on b1, Rook on c1), and the King's path (b1→g1) extends far beyond the Rook's position — passing through d1, e1, f1 etc. which may be occupied. makeMv() would directly overwrite those squares, effectively destroying own pieces. The fix now checks the union of the King path and the Rook path.

Clock-Expiry Handling v1.0.4 Rev35

Rev35 fixes a serious bug where clock expiry did not send a stop command. The original _onGameClockExpired() only set the gameOver state, without sending a stop command to the engine. The engine continued searching based on its internal wtime estimate, potentially far exceeding the GUI's 0-second mark. The fix: _onGameClockExpired() now calls AndroidBridge.engineStop() to immediately hard-stop the engine, and clears isAIThinking and the AI safety timer.

Rev35 critical fix: JS calls engineGoTimed() while measuring the clock, but by the time the "go" command is actually sent (after stopAndWaitForBestmove + ucinewgame + readyok waits, up to 6 seconds), the real clock has already decreased. The engine over-allocates search time based on the stale (too-large) wtime. Fix: engineGoTimed() now records the JS call timestamp and deducts the setup overhead from wtime/btime before sending "go", plus a safety margin (engineMoveOverhead, min 50ms) to ensure the engine's bestmove returns before the GUI clock hits zero.

Ponder Time Control Compliance v1.0.4 Rev31

The original startPonder() only sent "go ponder" without time parameters — violating the UCI spec. The UCI spec requires go ponder to include wtime/btime/winc/binc parameters so the engine can switch to normal timed search on receiving ponderhit. Without time parameters, either: (a) the engine searches indefinitely on ponderhit (no time pressure), or (b) the engine instantly returns bestmove on ponderhit (panic mode) — both abnormal for time control. Rev31 fix: startPonder() takes 4 new time parameters; JS passes the current gameClocks values. For non-time-control games, all-zeros are passed, and Java detects this and sends "go ponder" (without time parameters).

Chapter 16
Ponder Implementation Mechanism

The Ponder feature is implemented based on the go ponder command of the UCI (Universal Chess Interface) protocol. This chapter details the complete data flow, the four-layer protection mechanism, and time-control compliance.

Complete Data Flow

  1. Engine returns best move: the engine outputs bestmove e2e4 ponder e7e5 via stdout, where e2e4 is the recommended move and e7e5 is the engine's predicted opponent reply.
  2. Java-layer parsing: StockfishNative.processEngineLine() uses the regex ^bestmove\s+(\S+)(?:\s+ponder\s+(\S+))? to parse the ponder move, storing it in the _lastPonderMove field. If no ponder field is present, it is set to null.
  3. JS-layer obtains ponder data: both onBestMove() and onHintMove() obtain the current ponder move via AndroidBridge.getLastPonderMove(). getLastPonderMove() is a one-time read (auto-clears after read), preventing stale data from leaking into subsequent searches. The UCI move is temporarily stored in _pendingPonderMoveUCI, waiting for the first onPonderProgress() callback to convert to SAN format and assign to _ponderMoveSAN.
  4. Start ponder search: after the AI moves, onBestMove() applies the ponder move to the current position (simulating the opponent's move via makeMv()), generates a new FEN, and calls AndroidBridge.startPonder(fen). The Java layer sends position fen <FEN> and go ponder to the engine, entering infinite-search mode.
  5. Ponder progress callback and delayed conversion: while searching, the engine calls back onPonderProgress() in real time via info lines, updating _ponderBarInfo (search depth, node count, eval). On the first callback, if _pendingPonderMoveUCI exists, it is converted to SAN and assigned to _ponderMoveSAN (one-time, cleared after conversion). This delayed mechanism ensures the ponder move and search progress appear at the same time, avoiding incomplete UI display.
  6. Ponder Hit / Stop: when the player moves, executeMove() checks whether it matches the ponder move. On match, calls AndroidBridge.ponderHit() (engine uses existing search results to continue analysis); on mismatch, calls AndroidBridge.stopPonder() (engine stops searching).
  7. Display conditions: the AI Opponent Bar row 3 shows 🔮 ponder info only when both _ponderMoveSAN (SAN-format ponder move) and _ponderBarInfo (search progress) exist. When the AI is thinking, hint-loading, in review mode, or when the game ends, it is auto-hidden.

Four-Layer Protection Mechanism

Regalia uses four layers of protection to ensure Ponder data does not retain stale information:

  1. Generation Counter: the JS-layer core defense — all onPonderProgress() callbacks must pass generation validation before updating display data. Each new ponder session increments _ponderGen++; on callback, if _ponderGen !== _ponderStartGen, the callback is discarded.
  2. One-time read: getLastPonderMove() auto-clears after read, preventing old ponder moves from leaking into subsequent searches.
  3. Immediate DOM update: when clearing ponder variables, immediately calls _updateAIThinkDisplay() to avoid stale DOM values after variable clearing.
  4. Java-layer flag: the _discardingPonderBestmove flag prevents the post-ponder-stop bestmove from being incorrectly routed to the normal search processing flow.

Ponder Display in Hint Scenario

In the hint scenario, onHintMove() directly converts the ponder move to SAN and embeds it in hintText, displaying it alongside the recommended move (e.g. "🔦 Hint: Nf3 🔮 Ponder: e5") without waiting for Ponder search progress. Conditional display:

  • AI Hint bar: when the user requests a hint (💡 AI Hint), if the engine's bestmove is followed by ponder info, the hint text shows both the recommended move and the ponder move
  • AI Opponent Bar row 3: shows ponder info when the engine's background Ponder search is active
  • No ponder info: no Ponder-related content is shown, keeping the UI clean

Ponder Data Timeliness

Ponder info always comes from the ponder field of the most recent bestmove response. Each time the engine returns a new bestmove, the old ponder data is immediately cleared, and the display is updated only when the new bestmove contains ponder info, ensuring ponder data does not retain stale info. In the hint scenario, the ponder move is converted to SAN format when the engine returns bestmove (converted based on the position after the recommended move is executed, ensuring move correctness), directly embedded in the hint text without waiting for the Ponder search progress callback.

Anti-Stale-Data Mechanism (Multi-layer Protection)

  1. Each time ponder variables are cleared, immediately call _updateAIThinkDisplay() to update the DOM, avoiding stale DOM values after variable clearing
  2. onPonderProgress() adds a generation guard (_ponderGen / _ponderStartGen); if the current callback's generation differs from the start, ponder has been cleared by another flow and the callback should be discarded
  3. onEngineProgress() actively clears residual ponder data during AI thinking or hint loading
  4. Java layer adds the _discardingPonderBestmove flag, preventing the post-ponder-stop bestmove from being incorrectly routed to the normal search processing flow
  5. The hint area now requires both _ponderMoveSAN and _ponderBarInfo to display 🔮 ponder, avoiding stale ponder data being shown after a new ponder session starts but before the first progress frame arrives

Time Control Ponder Compliance v1.0.4 Rev31

The original startPonder() only sent "go ponder" without time parameters — violating the UCI spec. The UCI spec requires go ponder to include wtime/btime/winc/binc parameters so the engine can switch to normal timed search on receiving ponderhit. Rev31 fix: startPonder() takes 4 new time parameters; JS passes the current gameClocks values. For non-time-control games, all-zeros are passed, and Java detects this and sends "go ponder" (without time parameters). Fix: only set the flag and send stop when stateBefore != STATE_NONE.
